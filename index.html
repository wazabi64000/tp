Cours Complet sur les Bases de Données pour Débutants

Ce cours présente les concepts fondamentaux de la modélisation de données à travers trois approches complémentaires : SQL, MCD et UML. L'objectif est de comprendre comment créer et organiser les informations dans une base de données et comment ces informations peuvent être utilisées dans des applications réelles.

Nous prendrons comme exemple un site de petites annonces. Ce type de projet est simple à comprendre et permet de montrer de manière pratique :

  • Comment structurer les utilisateurs, annonces et catégories.
  • Comment relier les tables entre elles grâce aux clés étrangères.
  • Comment transformer un schéma conceptuel en SQL et en diagramme UML.

1. Comprendre les Tables SQL et les Relations

Une table en SQL représente une entité métier. Chaque ligne de la table correspond à un enregistrement (un objet réel ou conceptuel), et chaque colonne correspond à un attribut ou une propriété de cette entité.

Principes fondamentaux des tables

  • PRIMARY KEY : identifiant unique de chaque enregistrement. Aucune valeur ne peut se répéter.
  • FOREIGN KEY : permet de créer une relation entre deux tables, en reliant une colonne d'une table à la clé primaire d'une autre.
  • 1-N (un à plusieurs) : une entité peut être liée à plusieurs autres entités. Exemple : un utilisateur peut avoir plusieurs annonces.
  • N-M (plusieurs à plusieurs) : nécessite une table intermédiaire pour gérer la relation. Exemple : un étudiant peut suivre plusieurs cours, et chaque cours peut avoir plusieurs étudiants.
  • NOT NULL : indique que la valeur d'une colonne ne peut pas être vide.
  • AUTO_INCREMENT : permet à la base de données d’attribuer automatiquement un identifiant unique pour chaque nouvel enregistrement.

Exemple concret : site de petites annonces

Nous aurons trois entités principales : users, categories et annonces. 

CREATE TABLE users (
  id INT PRIMARY KEY AUTO_INCREMENT,
  email VARCHAR(255) NOT NULL,
  password VARCHAR(255) NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE categories (
  id INT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(100) NOT NULL
);

CREATE TABLE annonces (
  id INT PRIMARY KEY AUTO_INCREMENT,
  titre VARCHAR(255) NOT NULL,
  description TEXT,
  user_id INT NOT NULL,
  category_id INT NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(id),
  FOREIGN KEY (category_id) REFERENCES categories(id)
);

Explications :
- Chaque utilisateur possède un identifiant unique et peut créer plusieurs annonces.
- Chaque annonce appartient à une seule catégorie et à un seul utilisateur.
- Chaque catégorie peut contenir plusieurs annonces.
- On peut ajouter des champs supplémentaires pour un vrai projet : images, statut (active/sold), localisation, prix.

2. Modèle Conceptuel de Données (MCD)

Le MCD est une représentation abstraite des données. Il permet de visualiser les entités, leurs relations et les cardinalités sans se soucier du code SQL.

Entités pour notre exemple

  • Utilisateur : id, email, password, date de création
  • Annonce : id, titre, description, date de création, lien vers utilisateur et catégorie
  • Catégorie : id, nom de la catégorie

Relations et cardinalités

UTILISATEUR (1,N) ── crée ── (1,1) ANNONCE ANNONCE (1,1) ─ appartient ─ (1,N) CATEGORIE

Interprétation :
- Un utilisateur peut créer plusieurs annonces, mais chaque annonce a un seul utilisateur.
- Une annonce appartient à une seule catégorie, mais une catégorie peut contenir plusieurs annonces.
- Les cardinalités 1,N indiquent le nombre minimum et maximum d’objets impliqués dans la relation.

Adaptation à un vrai projet

  • Ajouter des relations supplémentaires si nécessaire, par exemple un système de commentaires : une annonce peut avoir plusieurs commentaires.
  • Ajouter des contraintes pour la validation des données : email unique, longueur minimale des champs.
  • Ajouter des attributs supplémentaires comme la date de publication ou la date de modification pour un suivi complet.

3. UML – Diagramme de classes simplifié

UML permet de représenter les objets d’un projet logiciel et leurs relations. Chaque classe correspond souvent à une table SQL dans un projet réel.

Explications :
- Les nombres 1 et N représentent les multiplicité dans la relation.
- Chaque annonce est liée à un utilisateur et à une catégorie.
- Dans un vrai projet, ces classes peuvent avoir des méthodes : publish(), delete(), update().

Adaptation à un projet réel

  • Utiliser un ORM (Object-Relational Mapping) comme Sequelize (Node.js) ou Doctrine (PHP) pour mapper ces classes aux tables SQL.
  • Ajouter des validations côté objet avant d’enregistrer dans la base.
  • Permettre l’extension facile des classes pour des fonctionnalités futures : notifications, favoris, filtres, etc.

4. Conseils pour débutants et mise en pratique

  • Commencez toujours par dessiner votre MCD pour visualiser les entités et relations.
  • Transformez ensuite le MCD en SQL pour créer les tables.
  • Utilisez UML pour planifier vos classes et leur logique métier.
  • Testez vos tables avec quelques données fictives pour vérifier les relations.
  • Pensez à ajouter des champs et contraintes réalistes pour un vrai projet : created_at, updated_at, status, image_url.

En suivant ces étapes, vous pourrez facilement adapter ce schéma simple de petites annonces à tout projet réel, que ce soit un site e-commerce, un blog, une application de gestion ou un réseau social.
